# TubeDL

Paste a YouTube link. Pick a quality (144p&ndash;4K). Download. Runs almost
entirely in your browser -- there's no traditional backend server, no
`yt-dlp` process, and no video ever touches a server you run or pay for.

## How it works

```
 ┌─────────────────────┐        ┌──────────────────────┐       ┌──────────────┐
 │  GitHub Pages        │  CORS  │  Cloudflare Worker    │       │  YouTube      │
 │  (static site)       │──────▶ │  (dumb relay, $0)     │──────▶│  InnerTube API│
 │                      │        │                       │       │  + video CDN  │
 │  - youtubei.js        │◀──────│  adds CORS headers,   │◀──────│               │
 │  - bgutils-js (PoToken)│      │  forwards bytes only  │       └──────────────┘
 │  - googlevideo (SABR) │        └──────────────────────┘
 │  - ffmpeg.wasm (mux)  │
 └─────────────────────┘
```

- **Frontend** (`frontend/`, built from `frontend-src/`): a static site.
  Parses the YouTube URL, talks to YouTube's InnerTube API to list
  available formats, downloads the chosen video/audio streams, and merges
  them into one file -- all client-side, using
  [`youtubei.js`](https://github.com/LuanRT/YouTube.js) (pure-JS InnerTube
  client), [`bgutils-js`](https://github.com/LuanRT/BotGuardUtils.js)
  (client-side PoToken/attestation), [`googlevideo`](https://github.com/LuanRT/googlevideo)
  (YouTube's SABR/UMP streaming protocol), and
  [`ffmpeg.wasm`](https://github.com/ffmpegwasm/ffmpeg.wasm) (in-browser
  muxing, no re-encoding).
- **Relay** (`worker/`): a ~100-line Cloudflare Worker. YouTube's API and
  CDN don't send permissive CORS headers to third-party origins, so
  something has to sit between the browser and YouTube purely to add those
  headers. It does no video processing and holds no secrets -- see
  "Why a relay at all?" below.

### Why a relay at all?

We tried hard to avoid this. The evidence (see the research this project
started from) is unambiguous: `youtubei.js`'s own maintainers ship a
mandatory CORS-relay reference implementation and their docs say outright
*"To use YouTube.js in the browser you must proxy requests through your own
server."* Google does not grant arbitrary third-party origins permission to
read response bytes from `googlevideo.com` or the InnerTube API -- and
merging separate video+audio streams (required for anything above 360p)
needs to read those bytes. A plain `<a download>` link can't get around
this either, because getting the actual stream URLs in the first place
already requires an InnerTube API call, which hits the same CORS wall.

The relay is the smallest possible concession to that reality: it's
free, serverless, does zero YouTube-specific logic, and everything that
actually matters (parsing formats, generating the PoToken, downloading,
merging) still happens on the user's own machine.

## Setup

1. **Deploy the relay** -- see [`worker/README.md`](worker/README.md).
   Takes about a minute, costs $0.
2. **Point the frontend at it**: edit `RELAY_BASE_URL` at the top of
   `frontend-src/src/app-entry.js` to your deployed relay URL.
3. **Enable GitHub Pages**: repo Settings &rarr; Pages &rarr; Source:
   "GitHub Actions". Push to `main` -- [`.github/workflows/deploy.yml`](.github/workflows/deploy.yml)
   builds `frontend-src` and publishes `frontend/` automatically.

## Local development

```bash
# Terminal 1 -- the relay
cd worker && npm install && npx wrangler dev --port 8787

# Terminal 2 -- build the frontend (set RELAY_BASE_URL to http://localhost:8787 first)
cd frontend-src && npm install && npm run build:dev

# Terminal 3 -- serve the static site
cd frontend && python3 -m http.server 8791
```

Then open `http://127.0.0.1:8791/`.

Run the extractor's fast Node-based regression check (bypasses the relay
entirely; no CORS in Node) with:

```bash
cd frontend-src && npm test
```

```

  index.html / css/     UI shell
  js/app.bundle.js       built output (generated by frontend-src, gitignored)

  src/extractor/         youtubei.js + googlevideo + bgutils-js integration
  src/mux.js              ffmpeg.wasm muxing
  src/app-entry.js        UI wiring (bundled into frontend/js/app.bundle.js)

worker/                 the Cloudflare Worker CORS relay
```

## Known limitation: some videos fail with "attestation required"

Short/typical videos work reliably end-to-end (verified repeatedly,
including via real-browser automated tests, at every resolution tier from
144p to 4K). However, some videos -- observed so far with longer/popular
ones (Big Buck Bunny 4K, Gangnam Style, Despacito), while e.g. Rick
Astley's "Never Gonna Give You Up" succeeds -- hit a hard server-side
`"attestation required"` error partway through downloading, even with a
correctly-generated, real (non-placeholder) PoToken, and even after
retrying with a completely fresh BotGuard session.

This isn't a bug we know how to fix client-side right now, and we're not
alone: `yt-dlp`'s own maintainers have an
[open, unmerged PR](https://github.com/yt-dlp/yt-dlp/pull/13515) adding
native SABR support to the most mature YouTube downloader in existence, and
their own notes tie some of these failures to a server-side experiment that
may require scraping signals from the actual watch-page HTML -- a
capability outside pure InnerTube API usage (what this project, and
`youtubei.js` itself, are built on). If you hit this, retrying sometimes
works; picking a lower quality tier sometimes works; there's currently no
guaranteed fix. PRs welcome if YouTube's side of this becomes better
understood.

## Legal

This tool doesn't check whether you have the right to download a given
video. That's on you as the operator/user. See the in-app footer. Public
YouTube-downloader sites have a real history of takedowns and lawsuits
(youtube-mp3.org, FLVTO/2conv, etc.) -- know what you're deploying before
you make this public.
